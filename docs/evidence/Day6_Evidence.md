### 2.2 列表页状态机

每个列表页现在支持四种完整状态：

```
加载中 → LoadingState（带旋转动画）
  ├─ 成功 + 有数据 → 列表
  │   ├─ 搜索命中 → 过滤列表
  │   └─ 搜索无结果 → EmptyState("未找到匹配结果")
  ├─ 成功 + 无数据 → EmptyState("暂无xxx信息")
  └─ 失败 → ErrorState（提示检查 JSON Server + 重新加载按钮）
```

| 失物招领 | 标题、物品名称、描述、地点 | `filteredItems` computed 属性 |
| 拼单搭子 | 标题、描述、地点、类型 | `filteredItems` computed 属性 |
| 跑腿委托 | 标题、描述、任务类型、出发地、目的地 | `filteredItems` computed 属性 |

搜索采用前端过滤：用户输入关键词后经过 300ms 防抖，前端按指定字段过滤列表。

## 6. 今日反思

**交互优化**不是"加动画"。今天最核心的收获是：列表页从两种状态（"有数据"/"没数据"）扩展为四种状态（加载中 / 有数据 / 无数据 / 请求失败），每种状态都给用户明确的视觉反馈。加载中的旋转动画告诉用户"我在工作"，错误状态的重新加载按钮给用户一个不刷新页面就能修复问题的途径。这种状态机思维是前端工程师的基本功——不能假设接口永远成功，也不能假设数据永远存在。

**加载状态**和**错误状态**是项目可展示性的关键。在 Day7 验收展示时，如果评审老师恰好没启动 JSON Server，项目会直接卡住或白屏。有了 ErrorState 和重新加载按钮，即使评审环境出问题，也能展示出项目的健壮性——这本身就是加分项。

搜索筛选虽然只用了简单的前端 `filter`，但配合 300ms 防抖和清除按钮，交互体验已经接近真实产品。更重要的是，这个实现方式（computed 属性 + 响应式关键字）是 Vue 最核心的编程模式，理解了这个模式，后续接任何真实 API 都能快速迁移。
