## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、originalPrice、condition、location、desc、specs | price 是交易核心信息，originalPrice 展示折扣力度，condition 描述成色，specs 用键值对灵活描述参数 |
| 失物招领 | lostFounds | title、type(lost/found)、itemName、category、location、eventTime、contactPhone、reward、desc | type 区分寻物/招领两种场景，itemName 描述物品，reward 激励归还，contactPhone 方便联系 |
| 拼单搭子 | groupBuys | title、type(ping/dazi/team)、category、targetCount、deadline、location、price、requirements、desc | targetCount 明确人数目标，deadline 控制截止时间，type 三种模式覆盖拼单/找搭子/组队 |
| 跑腿委托 | errands | title、taskType、category、reward、from、to、deadline、urgency、desc | from/to 描述空间路径，reward 激励接单，urgency 表示紧急程度，taskType 区分代买/代办/代取/代送 |

### 通用字段
所有发布类型共用以下字段：
- **标题**（必填）：信息的核心标识，最长50字
- **描述**（必填）：详细说明内容，最长500字
- **联系方式**（必填）：微信/QQ等，方便感兴趣的人联系
- **标签**（选填）：逗号分隔的关键词，便于筛选和搜索
- **图片**（选填）：最多9张，Base64本地预览

## 5. 人工调整内容

| 调整项 | 原因 |
|---|---|
| 创建 FormField.vue 可复用组件 | 减少 PublishView 中 20+ 个重复的表单项结构，提升代码可维护性 |
| 重构 PublishView.vue 模板使用 FormField | 将所有标准表单项替换为 `<FormField>` 组件，保留复杂交互（分类单选、类型切换按钮等）的原有结构 |
| 修正二手交易价格校验 | AI 初始使用 `< 0`（允许价格为 0），商品价格应该是"大于 0"而非"大于等于 0"，改为 `<= 0` |
| 修正跑腿委托报酬校验 | AI 初始使用 `<= 0`（不允许免费跑腿），改为 `< 0`（允许酬劳为 0，但不允许负数） |
| 移除 TradeFormData 中的 `favorited` 字段 | 该字段由 Omit 排除，不应在新建数据时传入，由后端/JSON Server 初始化 |
| 修复 parseSpecs 函数类型安全 | parts[0] 和 parts[1] 可能为 undefined，添加非空断言和空值过滤 |
| 修复 ItemCard.vue TypeScript 类型错误 | Options API computed 属性缺少返回类型标注，补充 `: string` 返回类型 |

## 7. 遇到的问题与解决方法

| 问题 | 解决方法 |
|---|---|
| PublishView.vue 模板中大量重复的 form-group 结构，约 20+ 处 | 创建 FormField.vue 组件封装 label + required + error + slot 模式，模板代码量减少约 30% |
| vue-tsc 报错 `'favorited' does not exist in type 'TradeFormData'` | TradeFormData 通过 Omit 排除了 favorited 字段，删除提交数据中的 `favorited: 0`，该字段由 JSON Server 初始化 |
| vue-tsc 报错 `Object is possibly 'undefined'` 在 parseSpecs 函数 | parts[0]/parts[1] 虽然在 length===2 保护下，但 TS 无法推断。使用非空断言 `!` 并增加空字符串过滤 |
| FormField 组件 slot 内元素的样式作用域问题 | Vue 3 中子组件根元素会获得父组件 scope 属性，且 slot 内容由父组件拥有，因此父组件 scoped CSS 对 slot 内容的样式规则仍然生效 |
| 切换发布类型后表单数据残留 | 通过 switchType() 函数在切换时先保存当前草稿，再执行 resetForm() 清空，最后尝试恢复新类型的草稿 |

## 8. 今日反思

**发布表单**是 Web 应用从"只读展示"走向"用户交互"的关键跨越。Day3 我们只是消费数据——从 JSON Server 获取列表、展示卡片；Day4 我们开始生产数据——用户通过表单输入信息，经过校验后写入数据库。这种从"消费者"到"生产者"的角色转变，是理解 CRUD 应用完整闭环的核心。一个好的发布表单不仅要收集足够的信息，还要考虑字段是否贴合业务场景：二手交易需要价格和成色，失物招领需要时间和地点，拼单需要人数和目标，跑腿需要路径和酬劳——每个字段背后都是真实的用户需求。

**表单校验**是前端不可推卸的责任。不能依赖"后端会校验"或者"用户会自觉填写"，前端必须在提交前拦截明显不合理的数据。Day4 实现的校验虽然基础（必填、数值范围），但覆盖了最常见的错误场景。更重要的是，校验失败后要给用户清晰的反馈——红色错误提示告诉用户"哪里错了、应该怎么改"，而不是一个笼统的"提交失败"。

**数据新增**让应用真正"活"了起来。通过 Axios 发送 POST 请求，用户发布的信息被 JSON Server 持久化到 db.json 中，刷新页面后数据依然存在。提交成功后的页面跳转和提示反馈，构成了一个完整的"输入→校验→提交→反馈→跳转"用户流程。这种端到端的体验闭环，是每个 Web 开发者必须掌握的基本功。

通过 FormField.vue 组件的封装，我体会到了组件抽象的实际价值：不是为了炫技，而是当同一个模式（label + input + error）在不同业务场景下重复出现 20+ 次时，抽象成组件不仅减少了代码量，更关键的是统一了交互行为——如果将来要调整错误提示的样式或必填标识的颜色，只需要改 FormField 一处即可。
